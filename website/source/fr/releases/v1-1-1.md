# v1.1.1

Date de sortie : `2026-04-19`

## Pourquoi cette version compte

`v1.1.1` rétablit la compatibilité de l'installeur lorsqu'un dépôt source expose encore soit l'arborescence legacy `src/plugin.yaml`, soit l'arborescence actuelle `plugin/plugin.yaml`.

## Ce qui a changé pour les utilisateurs

- la détection d'installation d'intégration accepte désormais les deux layouts auteur pendant les flux d'ajout
- les aliases first-party comme `gitlab` et `notion` cessent d'échouer simplement parce que le dépôt upstream utilise encore l'ancien arbre
- le layout actuel sous `plugin/` reste le chemin recommandé pour les nouveaux dépôts

## Ce que cela signifie en pratique

- `plugin-kit-ai add <source>` fonctionne de nouveau sur le catalogue first-party actuel
- les équipes n'ont plus besoin de savoir à l'avance quel layout auteur un dépôt source conserve encore
- la ligne publique de patch du CLI retrouve la compatibilité attendue par les exemples d'installation publiés

## Valeurs recommandées après v1.1.1

- gardez `plugin/plugin.yaml` comme layout canonique pour les nouveaux dépôts
- traitez `src/plugin.yaml` comme une entrée de compatibilité legacy, pas comme le nouveau chemin préféré
- relancez les installations first-party aliases ratées sur ce patch ou un plus récent

## Que faire maintenant

- passez à `v1.1.1` ou plus récent si `add gitlab` ou `add notion` échouait sur le CLI publié
- relancez le flux d'ajout si une installation précédente s'est arrêtée en cours de route
- conservez les nouveaux dépôts sur le layout d'authoring standard sous `plugin/`
