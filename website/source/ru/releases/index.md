## Выбор за 60 секунд

- Нужна актуальная рекомендация по поставке для Python или Node: читайте [v1.0.6](/ru/releases/v1-0-6).
- Нужен первый стабильный публичный baseline: читайте [v1.0.0](/ru/releases/v1-0-0).
- Нужен релиз Go SDK, который убрал лишнее friction для новых пользователей: читайте [v1.0.4 Go SDK](/ru/releases/v1-0-4-go-sdk).

## Выбор по задаче

- Планируете новый Python- или Node-проект и хотите понять актуальный default по поставке: начните с [v1.0.6](/ru/releases/v1-0-6).
- Оцениваете, достаточно ли проект зрелый для внедрения: сначала читайте [v1.0.0](/ru/releases/v1-0-0), потом [v1.0.6](/ru/releases/v1-0-6).
- Интегрируетесь через Go и хотите увидеть релиз, который упростил обычное использование SDK: читайте [v1.0.4 Go SDK](/ru/releases/v1-0-4-go-sdk).
- Обновляете существующий репозиторий и хотите понять, нужно ли менять текущий путь: начните с самой новой заметки и следуйте её migration callouts.

## Выбор по роли

- Автор плагина: читайте самый свежий релиз, который меняет setup defaults, starter guidance или delivery recommendations.
- Тимлид или владелец репозитория: сначала читайте самый свежий релиз и его migration callouts, если собираетесь закреплять путь сразу для нескольких repo.
- Интегратор или пользователь SDK: сразу переходите к релизу, который относится к вашему language surface, а потом следуйте его next-reading ссылкам.

## Какой релиз читать первым

- Читайте [v1.0.6](/ru/releases/v1-0-6), если вам важна актуальная история поставки для Python и Node.
- Читайте [v1.0.4 Go SDK](/ru/releases/v1-0-4-go-sdk), если вы интегрируетесь через Go и хотите увидеть релиз, который упростил обычное использование SDK.
- Читайте [v1.0.0](/ru/releases/v1-0-0), если хотите увидеть первый стабильный публичный baseline проекта.

## Что помогает решить этот раздел

- меняется ли рекомендуемый путь для вашего runtime или delivery model
- нужно ли мигрировать существующий репозиторий прямо сейчас или можно остаться на текущем пути
- на какой release note ссылаться внутри команды, когда спрашивают «что изменилось для пользователей?»

## Самый быстрый командный способ читать релизы

1. Владелец репозитория сначала открывает самый свежий релиз по своему пути.
2. Runtime owner проверяет там блоки «Что изменилось» и «Что не изменилось».
3. Команда ссылается на сам release note в планировании и миграции, а не спорит по памяти.

## Чем полезен раздел релизов

- помогает понять user-facing changes
- показывает, какая рекомендация стала сильнее или безопаснее
- даёт migration callouts до того, как вы меняете starter, runtime lane или delivery model
- помогает держать внутренние обсуждения привязанными к опубликованной guidance, а не к старым рабочим привычкам

## Что будет здесь

- Публичные release notes.
- Migration notes.
- Явные breaking-change callouts.
- Только user-facing changes.

Maintainer rehearsal и audit-материалы остаются вне публичного сайта.

## Чем этот раздел не является

- Это не лучший первый вход, чтобы учить продукт с нуля.
- Он не заменяет guide-страницы для выбора setup, target или runtime.
- Это правильное место после того, как путь уже выбран и нужно понять, что именно изменилось для пользователей.

## Рекомендуемый порядок чтения

1. Откройте самый свежий релиз, который относится к вашему runtime или delivery path.
2. Сначала прочитайте «Что это значит на практике» и migration callouts.
3. Возвращайтесь к guide или reference только после того, как станет ясно, изменилась ли сама рекомендация.
