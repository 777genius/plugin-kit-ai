# Внедрение в команду

Открывайте эту страницу, когда вопрос уже не в том, «может ли один человек заставить это работать», а в том, «сможет ли команда пользоваться одной моделью проекта, одними правилами репозитория и одной публичной guidance без путаницы».

## Выбор за 60 секунд

- Запускаете новый командный репозиторий: начните здесь, потом читайте [Плагин для команды](/ru/guide/team-ready-plugin).
- Стандартизируете существующий репозиторий: начните здесь, потом читайте [Миграцию существующей конфигурации](/ru/guide/migrate-existing-config).
- Раскатываете plugin-kit-ai сразу на несколько repo: начните здесь, потом читайте [Стандарт репозитория](/ru/reference/repository-standard) и [Интеграцию с CI](/ru/guide/ci-integration).
- Обновляете сразу несколько существующих repo после нового релиза или нового default: начните здесь, потом читайте [Плейбук обновлений и миграции](/ru/guide/upgrade-and-migration-playbook).

## Кому это особенно полезно

- тимлиду, владельцу репозитория или maintainer'у, которому задают вопрос «какой путь у нас официальный?»
- тем, кому нужен один публичный сценарий для setup, validation, CI и handoff
- тем, кто хочет остановить repo-by-repo импровизацию до того, как она превратится в командную folklore

## Путь внедрения

1. Осознанно выберите один поддерживаемый путь.
   Начинайте с самого узкого реального требования, а не обещайте все target’ы и runtime сразу.
2. Сделайте контракт репозитория видимым.
   Команда должна видеть authored state, generated outputs, выбранный target и главную команду проверки.
3. Превратите контракт в CI.
   `doctor`, `render` и `validate --strict` должны воспроизводиться на другой машине до того, как repo начнут считать здоровым.
4. Держите release guidance в контуре.
   Командные стандарты должны ссылаться на актуальный публичный release note, а не на старые сообщения в чате и не на память одного maintainer'а.

## Новый repo или существующий repo

- Новый repo:
  сначала выберите поддерживаемый путь, один раз создайте scaffold и закрепите стандарт репозитория до того, как команда начнёт копировать локальные привычки.
- Существующий repo:
  сначала свяжите текущие target files с authored source of truth, а потом планомерно двигайтесь к каноническому циклу render-and-validate.

## Как выглядит хорошее внедрение

- новый участник команды сразу видит, с какого guide начать
- repo явно показывает, какой target и runtime были выбраны
- CI воспроизводит generated outputs, а не доверяет вручную подправленным файлам
- release notes используются для проверки изменений guidance, а не придумываются заново внутри команды

## Чего делать не нужно

- закреплять путь по умолчанию, не сверившись с [Границей поддержки](/ru/reference/support-boundary)
- считать название starter’а долгосрочной границей репозитория
- позволять каждому repo придумывать собственный validation flow
- хранить критичные решения о setup только в Slack, чате или памяти ревьюера
