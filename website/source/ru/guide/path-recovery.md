# Восстановление пути

Открывайте эту страницу, когда repo ещё не сломан, но выбранный путь уже учит команду неправильной привычке, обещанию или операционной модели.

## Выбор за 60 секунд

- открывайте эту страницу, если repo работает, но выбранный путь явно не подходит для следующего этапа проекта
- открывайте её, если хотите исправить курс без драмы и без притворства, что repo надо выбросить целиком
- открывайте её до того, как неуверенный путь разойдётся дальше по другим repo и template
- не открывайте её для обычных обновлений, если healthy path остаётся тем же самым

## Что помогает решить эта страница

- нужен ли repo точечный разворот или более чистый перезапуск
- как вернуть repo на правильный путь, не теряя доверия к CI, generated outputs и командной guidance
- как удержать один reference repo чистым до более широкого rollout

## Безопасный шаблон восстановления

1. Назовите реальное несоответствие.
   Поймите, в чём именно ошибка: в starter, target, runtime или delivery model.
2. Заморозьте обещание.
   Перестаньте обещать поддержку сильнее той, которую текущий repo реально несёт.
3. Выберите одну чистую точку назначения.
   Сначала определите исправленный target, runtime или delivery model, и только потом трогайте код.
4. Восстанавливайте один reference repo.
   Не пытайтесь “чинить всё везде” одновременно.
5. Повторно прогоните канонический контракт.
   `doctor -> render -> validate --strict`
6. Только после этого обновляйте templates, CI и rollout guidance.

## Восстановление по типу ошибки

### Неправильный starter при правильной форме продукта

Если команда выбрала не тот starter, но сам target и runtime всё ещё верные, не нужно драматизировать. Starter — это входная точка, а не постоянная личность repo.

**Лучший ход:** нормализуйте repo к правильному managed layout и обновите starter guidance для следующих repo.

### Неправильный target

Если repo собран как runtime, хотя реальный продукт — это packaging, extension или владение workspace configuration, относитесь к этому как к смене формы продукта, а не как к маленькому patch.

**Лучший ход:** вернитесь к [Выбору target](/ru/guide/choose-a-target), сначала исправьте форму выхода, а уже потом снова прогоните render и строгую проверку.

### Неправильный runtime

Если выбрали Node или Python, хотя реальная долгосрочная потребность сводилась к самому сильному production default, восстановление обычно дешевле всего до того, как repo разошёлся дальше.

**Лучший ход:** сравните это с [Выбором runtime](/ru/concepts/choosing-runtime), проверьте, оправдана ли операционная цена, и если нет — вернитесь к более сильному default до rollout.

### Неправильная модель поставки

Если shared runtime package появился слишком рано или vendored helpers уже мешают reuse across repos, исправляйте delivery model явно, а не наслаивайте новые исключения поверх старых.

**Лучший ход:** перепроверьте [Выбор модели поставки](/ru/guide/choose-delivery-model) и доведите один reference repo до чистого состояния до любой командной стандартизации.

## Когда чинить, а когда начинать заново

Чините текущий repo, когда:

- форма продукта всё ещё выбрана правильно
- generated outputs по-прежнему отражают один понятный source of truth
- команда может объяснить коррекцию одной короткой публичной фразой

Начинайте с более чистого baseline заново, когда:

- выбранный путь уже учит команду неправильной форме продукта
- вручную подправленные generated files скрывают реальное authored state
- CI доказывает только локальную удачу, а не реальный managed contract
- repo собираются превратить в образец для следующих repo

## Что нужно защитить во время восстановления

- одно публичное объяснение исправленного пути
- один reference repo, который чисто доказывает новый baseline
- один CI contract, совпадающий с authored source of truth
- одну связанную release или policy note, если изменение влияет на командный стандарт

## Чего не делать

- не наслаивайте новые исключения поверх пути, который вы уже признали неправильным
- не раскатывайте сомнительное исправление на несколько repo до того, как один reference repo стал чистым
- не относитесь к generated drift как к косметике во время восстановления
- не обещайте поддержку сильнее только потому, что repo “пока ещё работает”
