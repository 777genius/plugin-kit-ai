# Сигналы drift baseline

Открывайте эту страницу, когда repo всё ещё выглядит здоровым, но есть ощущение, что реальный командный baseline и фактическое поведение repo уже начинают расходиться.

## Выбор за 60 секунд

- откройте эту страницу, если CI зелёный, но repo уже труднее объяснять, чем раньше
- откройте её, если один repo всё ещё “работает”, но новые repo уже не понимают, что с него копировать
- откройте её до того, как объявить repo reference baseline для остальных
- не используйте её вместо обычного bug fixing; она нужна там, где проблема уже в drift стандарта

## Что помогает решить эта страница

- начал ли repo уходить от объявленного стандарта
- продолжает ли команда учить один чистый baseline
- в какой момент пора перестать считать repo “достаточно здоровым” и поправить курс

## Самая быстрая проверка drift

Считайте baseline уплывающим, если уже выполняются хотя бы два пункта:

- CI ещё проходит, но другой maintainer не может объяснить выбранный путь без частной помощи
- generated outputs воспроизводятся только после локальных привычек или недокументированных шагов
- README, release note и support policy уже не указывают на один и тот же текущий путь
- repo стал “особым случаем”, который другим repo советуют не копировать напрямую
- команда всё чаще спрашивает, является ли reference repo вообще ещё стандартом

## Типичные сигналы drift

### 1. Repo проходит CI, но не проходит проверку на объяснимость

Если другой maintainer может клонировать repo и пройти CI, но всё ещё не понимает, почему были выбраны именно этот starter, target, runtime или delivery model, baseline уже уходит в drift.

### 2. Rendered outputs требуют приватных правок

Если `render` формально работает, но команда негласно ожидает ручные правки или локальную чистку после генерации, repo уже не доказывает публичный контракт чисто.

### 3. Reference repo стал “особым” repo

Если всё чаще звучит фраза “делайте как в том repo, только вот эти три исключения не копируйте”, значит reference repo уже учит не baseline, а исключениям.

### 4. Публичная guidance и реальный repo перестали совпадать

Если release notes, support promises, repo docs и CI намекают на чуть разные пути, команда уже живёт не по одному стандарту, а по drift.

### 5. Новые repo стартуют с другой mental model

Если новые repo всё чаще выбирают другие starters, targets или delivery paths, потому что текущий стандарт уже неясен, проблема не в онбординге. Ослаб сам baseline.

## Что проверить первым

- может ли новый maintainer понять выбранный путь только по публичным docs?
- доказывает ли `doctor -> render -> validate --strict` intended path без локального фольклора?
- совпадает ли repo с тем support promise, которое команда публично обещает?
- reference repo всё ещё достаточно прост, чтобы его копировать, или он уже только “старый и привычный”?

## Что делать, когда замечен drift

1. Перестаньте продвигать repo как чистый baseline.
2. Определите, что именно уплыло: объяснение, support promise или сам path.
3. Сначала исправьте один reference repo.
4. Обновите CI, repo docs и публичную guidance вместе.
5. Только после этого возобновляйте rollout и обновление templates.

Baseline начал уходить в drift в тот момент, когда команда доверяет ему уже по привычке, а не по воспроизводимому доказательству.
