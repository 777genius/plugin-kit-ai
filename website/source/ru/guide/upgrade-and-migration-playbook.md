# Плейбук обновлений и миграции

Открывайте эту страницу, когда у команды уже есть живые репозитории и главный вопрос звучит так: как принять новую guidance, новые defaults или managed project model так, чтобы не потерять доверие и не устроить хаос.

## Выбор за 60 секунд

- Обновляете существующий managed repo: начните здесь, потом читайте самый свежий подходящий [release note](/ru/releases/).
- Переводите native target files в managed project model: начните здесь, потом читайте [Миграцию существующей конфигурации](/ru/guide/migrate-existing-config).
- Раскатываете новый default сразу на несколько repo: начните здесь, потом читайте [Внедрение в команду](/ru/guide/team-adoption) и [Готовность к продакшену](/ru/guide/production-readiness).
- Раскатываете новый default сразу на несколько repo: начните здесь, потом читайте [Внедрение в команду](/ru/guide/team-adoption), [Готовность к продакшену](/ru/guide/production-readiness) и [Rollout на уровне команды](/ru/guide/team-scale-rollout).

## Что помогает решить этот playbook

- нужен ли repo просто version bump, смена delivery path или уже более глубокая миграция
- как раскатить новую guidance без того, чтобы каждый repo трактовал её по-своему
- в какой момент нужно остановиться и ещё раз проверить support boundary до обещаний downstream-пользователям

## Безопасный шаблон обновления

1. Прочитайте самый свежий release note, который относится к вашему runtime или delivery path.
2. Разделите «что изменилось» и «что не изменилось».
3. Определите, что происходит с этим repo:
   - тот же путь, но с новыми defaults
   - смена delivery model
   - миграция из native config в managed project model
4. Повторно прогоните канонический контракт:
   `doctor -> render -> validate --strict`
5. Зафиксируйте изменение в CI и документации repo до того, как считать обновление завершённым.

## Чтение по сценарию

- Существующий Go repo без смены пути:
  прочитайте подходящий release note, проверьте изменения SDK или CLI, затем заново прогоните стандартный validation flow.
- Существующий Python- или Node-repo:
  сначала прочитайте свежую guidance по delivery, особенно если релиз меняет рекомендации вокруг `--runtime-package`.
- Существующий native-config repo:
  относитесь к изменению как к миграции в managed project model, а не как к маленькому patch.
- Несколько repo в одной команде:
  сначала выберите один reference repo, проверьте новый путь на нём, а уже потом раскатывайте дальше через [Rollout на уровне команды](/ru/guide/team-scale-rollout).

## Что проверить до любого изменения

- остаётся ли выбранный target внутри публичной границы поддержки
- действительно ли релиз меняет рекомендуемый default, а не только уточняет существующую guidance
- не держится ли repo уже сейчас на вручную подправленных generated files
- доказывает ли CI authored source of truth или только локальное состояние одного разработчика

## Хорошая дисциплина миграции

- сначала обновляйте один repo, а не все repo параллельно
- держите release notes прямо в migration plan
- обновляйте starter или внутренние repo templates только после того, как один реальный repo чисто прошёл новый путь
- относитесь к generated drift как к сигналу, а не как к косметике

## Чего делать не нужно

- не раскатывайте новый default только потому, что он «звучит новее», не сверившись с support и release notes
- не считайте миграцию из native config маленькой косметической чисткой
- не стандартизируйте новый runtime или delivery path, не обновив CI и документацию репозитория
- не читайте один release note как обещание одинаковой поддержки для всех language и target paths

## Финальное правило

Обновление завершено только тогда, когда другой участник команды может клонировать repo, воспроизвести generated outputs, пройти `validate --strict` и объяснить новый выбранный путь по публичным docs и release notes.
