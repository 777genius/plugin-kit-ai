# Антипаттерны выбора

Используйте эту страницу, когда repo уже технически работает, но выбранный путь ощущается слишком тяжёлым, слишком дорогим в сопровождении или его слишком трудно объяснять другим.

## Выбор за 60 секунд

- откройте эту страницу, если repo работает, но выбранный путь уже ощущается тяжелее самой задачи
- откройте эту страницу, если команда снова и снова объясняет одни и те же исключения в чатах и PR
- откройте эту страницу до того, как стандартизировать starter, target или delivery model на несколько repo
- не открывайте её ради лишней оптимизации здорового repo, который уже соответствует реальному требованию

## Что помогает решить эта страница

- выбрали ли вы не тот starter, target, runtime или delivery model
- пора ли перезапускать repo с более чистой стартовой точки
- не скрывается ли за “технической проблемой” обычная путаница в форме продукта

## Самый быстрый smell test

С большой вероятностью путь выбран неправильно, если уже выполняются хотя бы два пункта:

- repo технически работает, но команде всё время приходится объяснять, почему он устроен именно так
- первое реальное требование уже не совпадает с текущим target
- package, extension или workspace outputs ожидают как runtime plugins
- Node или Python были выбраны, хотя реальная потребность сводилась к самому сильному production default
- shared runtime package появился раньше, чем в команде появился один чистый reference repo
- rollout уже идёт на несколько repo, хотя по-прежнему спорят о starter, target, runtime или delivery model

## Типичные антипаттерны

### 1. Package или workspace targets воспринимают как runtime path

Это самый частый тип путаницы.

Если итогом должен быть package, extension или workspace configuration, не ждите от него поведения исполняемого runtime plugin. Repo может быть собран правильно, но ожидания направлены в неверную форму продукта.

**Что делать:** вернитесь к [Выбору target](/ru/guide/choose-a-target), [Package и workspace targets](/ru/guide/package-and-workspace-targets) и [Обещаниям поддержки по путям](/ru/reference/support-promise-by-path).

### 2. Node или Python выбрали там, где на самом деле нужен был самый сильный default

Node и Python — это реальные поддерживаемые пути. Они не являются ошибкой сами по себе.

Ошибкой они становятся тогда, когда команде не нужна языковая специфика, а на деле требовался просто самый сильный и наименее хрупкий production path. В таком случае repo берёт на себя больше runtime-ответственности, чем нужно.

**Что делать:** перепроверьте [Выбор runtime](/ru/concepts/choosing-runtime) и [Готовность к продакшену](/ru/guide/production-readiness). Если ничего по-настоящему не требует Node или Python, возвращайтесь к Go default до того, как rollout уйдёт дальше.

### 3. Название starter’а принимают за постоянную границу repo

Starter показывает, **как начать**, а не **чем проект обязан остаться навсегда**.

Если команда читает название Claude или Codex starter’а как доказательство того, что repo никогда не сможет вырасти дальше первого пути, значит сама mental model уже съехала.

**Что делать:** перечитайте [Один проект, несколько target’ов](/ru/guide/one-project-multiple-targets) и [Managed Project Model](/ru/concepts/managed-project-model).

### 4. Shared runtime package выбирают слишком рано

`plugin-kit-ai-runtime` — это поддерживаемый путь, а не запасной вариант. Но это всё равно неверный первый ход, если у команды ещё нет одного чистого reference repo.

Если каждый repo всё ещё ищет собственную форму, shared runtime package добавляет координационную цену раньше времени.

**Что делать:** начните с vendored helpers или с самого сильного Go default, а shared runtime package вводите только после появления одного здорового эталонного repo. Вернитесь к [Выбору модели поставки](/ru/guide/choose-delivery-model).

### 5. CLI wrappers принимают за runtime API

Если npm или PyPI wrappers читают как публичные runtime surfaces, repo уже смешивает установку и исполнение в одну сущность.

**Что делать:** вернитесь к [Установке](/ru/guide/installation), [Каналам установки](/ru/reference/install-channels) и [Обзору API](/ru/api/).

### 6. Путь стандартизируют до проверки обещания поддержки

Команды иногда стандартизируют путь просто потому, что он “один раз сработал”, а уже потом обнаруживают, что сила обещаний, операционная цена и масштабируемость были совсем не теми, что они предполагали.

**Что делать:** проверяйте [Границу поддержки](/ru/reference/support-boundary), [Обещания поддержки по путям](/ru/reference/support-promise-by-path) и [Поддержку target’ов](/ru/reference/target-support) до того, как превращать эксперимент в командную политику.

### 7. Rollout масштабируют до того, как один reference repo стал чистым

Если rollout уже пошёл на несколько repo, а команда всё ещё спорит про starter, target, runtime или delivery model, значит она масштабирует не стандарт, а путаницу.

**Что делать:** остановите rollout, доведите один reference repo до чистого состояния, а затем возвращайтесь к [Внедрению в команду](/ru/guide/team-adoption), [Rollout на уровне команды](/ru/guide/team-scale-rollout) и [Плейбуку обновлений и миграции](/ru/guide/upgrade-and-migration-playbook).

## Когда начинать заново, а когда нет

Перезапускайте путь с более чистой точки, когда:

- выбранный путь требует слишком много повторных объяснений
- обещание поддержки не совпадает с реальной операционной потребностью
- форма repo уже учит команду неправильной mental model

Не начинайте заново только потому, что другой путь выглядит красивее на бумаге. Перезапуск нужен тогда, когда текущий выбор усложняет эксплуатацию repo, передачу другим людям или масштабирование.

## Финальное правило

Правильный путь — это не тот, который просто работает сегодня. Это тот путь, который по-прежнему легко объяснить, проверить, передать другому человеку и масштабировать, когда repo перестанет быть одиночным экспериментом.
