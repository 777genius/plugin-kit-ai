# Политика здоровых исключений

Открывайте эту страницу, когда repo действительно должен отличаться от остальных, но команде нужен публичный критерий, когда это отличие ещё оправдано, а не превращается в вечную folklore.

## Выбор за 60 секунд

- откройте эту страницу, если кто-то говорит «этот repo особенный», и нужно понять, здорово это или нет
- откройте её до того, как превратить один repo в постоянный класс исключений
- откройте её, если команда продолжает обрастать оговорками вокруг одного repo вместо исправления пути
- не используйте её как оправдание слабому стандарту; она нужна, чтобы исключения оставались узкими и честными

## Что помогает решить эта страница

- является ли repo оправданным исключением или уже нездоровым drift
- насколько узким должен оставаться special-case path
- когда исключение нужно нормализовать, заменить или убрать

## Как выглядит здоровое исключение

Здоровое исключение:

- названо явно
- узкое по области применения
- объяснено в публичных docs или docs самого repo
- не выходит за публичную support boundary
- всё ещё проходит понятный validation contract

Если исключение требует частного устного комментария, оно уже начинает становиться нездоровым.

## Как выглядит нездоровое исключение

Нездоровое исключение обычно проявляется так:

- repo “особенный” только потому, что его никто не хочет исправлять
- команда уже не может внятно объяснить, зачем это исключение всё ещё существует
- CI, docs и release guidance не совпадают в описании реального правила
- новые repo случайно копируют это исключение
- исключение тихо меняет support promise, не будучи объявленным

## Хорошие причины для исключения

- у одной платформы или target’а есть реальное product-specific ограничение
- repo сознательно закрывает временную фазу миграции
- repo доказывает узкий beta path и честно не притворяется новым default
- команде нужно одно временное исключение, пока готовится более чистый baseline

## Плохие причины для исключения

- «один раз же уже сработало»
- «переписывать неудобно»
- «это понимает только один maintainer»
- «docs это пока не покрывают чисто»

Это сигналы к исправлению baseline, а не причины благословлять drift.

## Тест исключения

Считайте исключение здоровым только если на все вопросы ниже ответ — “да”:

1. Исключение явно задокументировано?
2. Исключение уже и уже, чем командный default?
3. Repo всё ещё проходит понятный контракт `doctor -> render -> validate --strict`?
4. Исключение сохраняет публичное support promise, а не тихо расширяет его?
5. Новые repo понимают, что не должны его копировать без такого же реального ограничения?

## Что делать со здоровым исключением

- держать его узким
- один раз задокументировать причину
- ссылаться на точную support или migration note, которая его объясняет
- регулярно пересматривать, нужно ли оно всё ещё после смены baseline

## Что делать с нездоровым исключением

- перестать считать его допустимым стандартом
- решить, нужно ли исправление, восстановление или вывод из использования
- вернуть команду ближе к чистому baseline
- не позволять templates и starter guidance копировать это дальше

## Финальное правило

Special-case repo остаётся здоровым только тогда, когда команда может объяснить, почему он исключение, чисто доказать это и не дать другим repo копировать его случайно.
