# v1.1.1

发布日期：`2026-04-19`

## 为什么这个版本重要

`v1.1.1` 恢复了安装器兼容性，即使源仓库仍然暴露 legacy 的 `src/plugin.yaml` 树，或者当前的 `plugin/plugin.yaml` 树。

## 对用户有什么变化

- 集成安装检测现在会在 add flow 中接受这两种 authored layout
- `gitlab`、`notion` 之类的 first-party aliases 不会再因为上游仓库还保留旧树而失败
- 当前基于 `plugin/` 的 layout 仍然是新仓库的推荐路径

## 这在实践中意味着什么

- `plugin-kit-ai add <source>` 在当前 first-party 目录上重新可用了
- 团队不再需要提前知道某个源仓库仍然保留哪一种 authored layout
- 公开 CLI patch 线重新获得了已发布安装示例所承诺的兼容性

## v1.1.1 之后的推荐默认值

- 对新仓库继续使用 `plugin/plugin.yaml` 作为规范 layout
- 将 `src/plugin.yaml` 视为 legacy 兼容输入，而不是新的首选路径
- 在这个补丁或更高版本上重新运行失败的 first-party alias 安装

## 现在该做什么

- 如果已发布 CLI 上的 `add gitlab` 或 `add notion` 失败，请升级到 `v1.1.1` 或更高版本
- 如果之前的安装中途停止，请重新运行 add flow
- 让新仓库继续留在标准的 `plugin/` authoring layout
