# v1.1.2

Fecha de lanzamiento: `2026-04-19`

## Por qué importa esta versión

`v1.1.2` corrige la parte de Gemini de los flujos completos de add multi-target cuando la fuente proviene de un GitHub repo path o de un first-party alias que termina resolviéndose allí.

## Qué cambió para los usuarios

- las instalaciones Gemini para fuentes GitHub repo-path ahora pasan por el flujo administrado de proyección local
- first-party aliases como `gitlab` y `notion` ya no rompen las ejecuciones completas de `add` por culpa de la rama Gemini
- la historia de instalación multiagente administrada vuelve a permanecer alineada entre los targets compatibles

## Qué significa esto en la práctica

- `plugin-kit-ai add gitlab` y `plugin-kit-ai add notion` vuelven a completarse en la línea de parches actual
- los equipos pueden seguir usando la ruta simple de first-party aliases sin workarounds específicos de Gemini
- la línea base pública actual ya cubre tanto la compatibilidad de authored layouts como la corrección repo-path de Gemini

## Valores recomendados después de v1.1.2

- siga usando first-party aliases como la ruta pública normal de instalación cuando el catálogo ya los ofrezca
- mantenga las instalaciones administradas de repo-path en la línea de parches actual o una posterior
- si Gemini falló antes, vuelva a ejecutar el flujo completo de add multi-target en `v1.1.2` o posterior

## Qué hacer ahora

- actualice a `v1.1.2` o posterior si las instalaciones multi-target fallaban durante el paso de Gemini
- vuelva a ejecutar el flujo de add para los first-party aliases afectados
- trate `v1.1.2` como la línea base pública actual para la guía de instalación y los ejemplos
