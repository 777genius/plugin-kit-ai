# v1.1.1

Fecha de lanzamiento: `2026-04-19`

## Por qué importa esta versión

`v1.1.1` restaura la compatibilidad del instalador cuando un repositorio fuente todavía expone el árbol heredado `src/plugin.yaml` o el árbol actual `plugin/plugin.yaml`.

## Qué cambió para los usuarios

- la detección de instalación de integración ahora acepta ambos layouts authored durante los flujos de add
- first-party aliases como `gitlab` y `notion` dejan de fallar solo porque el repositorio upstream todavía usa el árbol antiguo
- el layout actual bajo `plugin/` sigue siendo la ruta recomendada para los repositorios nuevos

## Qué significa esto en la práctica

- `plugin-kit-ai add <source>` vuelve a funcionar en el catálogo first-party actual
- los equipos ya no necesitan saber de antemano qué layout authored mantiene un repositorio fuente
- la línea pública de parches del CLI recupera la compatibilidad que prometen los ejemplos publicados de instalación

## Valores recomendados después de v1.1.1

- mantenga `plugin/plugin.yaml` como el layout canónico para los repositorios nuevos
- trate `src/plugin.yaml` como entrada de compatibilidad heredada, no como la ruta preferida para nuevos proyectos
- vuelva a ejecutar las instalaciones fallidas de first-party aliases con este parche o uno posterior

## Qué hacer ahora

- actualice a `v1.1.1` o posterior si `add gitlab` o `add notion` falló en el CLI publicado
- vuelva a ejecutar el flujo de add si una instalación anterior se quedó a medio camino
- mantenga los repositorios nuevos en el layout estándar de authoring bajo `plugin/`
