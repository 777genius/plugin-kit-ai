# codex-python-local

Codex executable project generated by [plugin-kit-ai](https://github.com/plugin-kit-ai/plugin-kit-ai) `init`.

This local plugin starter is the fastest path when you want repo-local integration and quick iteration in `python`.
This repo-local reference example is a fast-start lane for `codex-runtime`. For the stronger Go runtime lane, see [../../plugins/codex-basic-prod](../../plugins/codex-basic-prod).

## Runtime

- Runtime: `python`
- Entrypoint: `./bin/codex-python-local`
- Execution mode: `launcher`
- Status: `public-beta`, repo-local executable ABI

## Bootstrap Contract
- `venv`, `requirements.txt`, and `uv` expect repo-local `.venv`
- `poetry` and `pipenv` can validate through manager-owned envs
- If `.venv` exists but is broken, `plugin-kit-ai validate --strict` will fail until you recreate it
- `plugin-kit-ai install` does not manage Python dependencies or virtualenv creation

## First Run
```bash
plugin-kit-ai doctor .
plugin-kit-ai bootstrap .
plugin-kit-ai validate . --platform codex-runtime --strict
./bin/codex-python-local notify '{"client":"codex-tui"}'
```
On Windows, repo-local env flows still use `.venv\\Scripts\\python.exe`; `poetry` and `pipenv` may validate against manager-owned envs without creating `.venv`.

## Stable Default

- `Notify`

The scaffold wires the documented stable Codex `notify` path for the `codex-runtime` lane.
Treat `plugin-kit-ai validate --strict` as the CI-grade readiness gate for this runtime lane.
This path does not add managed dependency installation or packaged distribution for interpreted runtimes.

## Target Files

- `targets/codex-runtime/package.yaml`: authored Codex runtime metadata
- `.codex/config.toml`: rendered managed Codex config

Keep stdout reserved for Codex responses and write diagnostics to stderr only.
