# codex-node-typescript-local

Codex executable project generated by [plugin-kit-ai](https://github.com/777genius/plugin-kit-ai) `init`.

This local plugin starter is the fastest TypeScript path when you want repo-local integration and quick iteration in `node`.
This repo-local reference example is a stable supported lane for `codex-runtime`, not a second-class fallback.
For the more self-contained Go path, where downstream users run a compiled binary without a separately installed Node runtime, see [../../plugins/codex-basic-prod](../../plugins/codex-basic-prod).

## Runtime

- Runtime: `node` with `--typescript`
- Entrypoint: `./bin/codex-node-typescript-local`
- Execution mode: `launcher`
- Status: `public-stable`, repo-local local-runtime subset

## Bootstrap Contract
- Runtime requirement: system Node.js `20+` must be installed on the machine running the plugin
- Helper layer: `src/plugin-runtime.ts`
- Install manager: lockfile-first manager detection, falling back to `npm`
- TypeScript output: `dist/main.js`
- `plugin-kit-ai bootstrap .` runs install first and then `build`

## First Run
```bash
plugin-kit-ai doctor .
plugin-kit-ai bootstrap .
plugin-kit-ai validate . --platform codex-runtime --strict
./bin/codex-node-typescript-local notify '{"client":"codex-tui"}'
```

## Stable Default

- `Notify`

The scaffold wires the documented stable Codex `notify` path for the `codex-runtime` lane.
Treat `plugin-kit-ai validate --strict` as the CI-grade readiness gate for this runtime lane.
This path does not add packaged distribution for interpreted runtimes.
Choose the Go lane instead when you want to avoid telling plugin users to install Node first.

## Target Files

- `src/targets/codex-runtime/package.yaml`: authored Codex runtime metadata
- `.codex/config.toml`: generated managed Codex config
- `src/plugin-runtime.ts`: official helper API for `onNotify`

Keep stdout reserved for Codex responses and write diagnostics to stderr only.
