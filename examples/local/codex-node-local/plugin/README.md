# codex-node-local

Codex executable project generated by [plugin-kit-ai](https://github.com/777genius/plugin-kit-ai) `init`.

This local plugin starter is the fastest path when you want repo-local integration and quick iteration in `node`.
This repo-local reference example is a fast-start lane for `codex-runtime`. For the stronger Go runtime lane, see [../../plugins/codex-basic-prod](../../plugins/codex-basic-prod).
For the TypeScript-first variant, see [../codex-node-typescript-local](../codex-node-typescript-local).

## Runtime

- Runtime: `node`
- Entrypoint: `./bin/codex-node-local`
- Execution mode: `launcher`
- Status: `public-stable`, repo-local local-runtime subset

## Bootstrap Contract
- Runtime: system Node.js `20+`
- Commit `package-lock.json` or an equivalent deterministic lockfile for production repos
- JavaScript is the default scaffold; use `plugin-kit-ai init ... --runtime node --typescript` for the first-class TypeScript authoring path
- TypeScript-shaped Node projects build to `dist/`; `plugin-kit-ai bootstrap .` runs both `npm install` and `npm run build` before `validate --strict`

## First Run
```bash
plugin-kit-ai doctor .
plugin-kit-ai bootstrap .
plugin-kit-ai validate . --platform codex-runtime --strict
./bin/codex-node-local notify '{"client":"codex-tui"}'
```

## Stable Default

- `Notify`

The scaffold wires the documented stable Codex `notify` path for the `codex-runtime` lane.
Treat `plugin-kit-ai validate --strict` as the CI-grade readiness gate for this runtime lane.
This path does not add managed dependency installation or packaged distribution for interpreted runtimes.

## Target Files

- `plugin/targets/codex-runtime/package.yaml`: authored Codex runtime metadata
- `.codex/config.toml`: generated managed Codex config

Keep stdout reserved for Codex responses and write diagnostics to stderr only.
